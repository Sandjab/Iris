# Phase 4.x cleanup — Design

> Date: 2026-05-27
> Status: ready for implementation
> Scope: déblayer les 3 RPC restants des stubs `exit 64` (Phase 5.1), sans toucher au store SQLite (différé à une éventuelle Phase 4.y dédiée).

## 1. Objectif

Boucler le contrat admin défini en SPECS §13.2 sur trois points laissés en attente :

- **`rule.add/list/delete`** — gestion runtime des hôtes MITM whitelistés.
- **`config.reload`** — recharger la configuration TOML à chaud (RPC + SIGHUP).
- **`events.clear`** — vider le buffer in-memory des events.

Le store SQLite des events reste hors scope (CLAUDE.md §11 décision ouverte).

## 2. Contraintes

- CLAUDE.md §6 — invariants sécurité non-négociables (pas de leak secret via les nouveaux endpoints).
- CLAUDE.md §1 — aucune nouvelle dépendance.
- SPECS §13.2 — wire format JSON-RPC déjà fixé.
- Phases 0-5.3.1 verrouillées : ne pas refactor au passage.

## 3. Décisions clés

### 3.1 Persistance des rules ajoutées via RPC : overlay JSON séparé

**Choix** : fichier `runtime-rules.json` à côté du `admin_socket` configuré (donc dérivé du même répertoire, typiquement `~/Library/Application Support/iris/runtime-rules.json`).

**Sémantique overlay** :
- `rule.list` retourne `TOML ∪ JSON` (union dédupliquée par host).
- `rule.add(host)` :
  - Si déjà dans TOML : no-op idempotent, retourne la `MITMRule` synthétisée (createdAt = epoch 0 ou Date du fichier TOML — voir §3.4).
  - Si déjà dans JSON : no-op idempotent.
  - Sinon : append au JSON + atomic write.
- `rule.delete(host)` :
  - Si dans TOML : refuse — erreur JSON-RPC custom `-32010 ruleProtected` avec message « rule defined in config TOML, edit the file directly ». Fail-loud, cohérent CLAUDE.md §12.
  - Si dans JSON : retire + atomic write, retourne `{ deleted: true }`.
  - Si absent partout : erreur `-32011 ruleNotFound`.

**Atomic write** : `write(tempPath)` → `rename(tempPath, finalPath)`. Permissions `0600`. Pattern déjà utilisé en `MCPCommands` (Phase 5.3) pour `.iris.bak` — réutilisable au pattern près, mais une logique dédiée dans `RuntimeRulesStore` actor évite la dépendance au CLI.

**Pourquoi pas réécrire le TOML** : si l'utilisateur édite manuellement son `config.toml` (commentaires, formatage, ordre des sections), une réécriture programmatique écrase ses choix. Le JSON est runtime-mutable, le TOML reste source-of-truth user-edited.

**Pourquoi pas in-memory only** : UX surprenante (ajout perdu silencieusement au restart). Cassé pour le menu-bar Phase 6 qui présupposera des mutations persistantes.

### 3.2 Hot-reload `config.reload` : partition stricte

**Champs hot-reloadables** (re-appliqués à chaud sur le `ProxyServer` existant) :
- `security.on_exfil_attempt`
- `security.max_substitutions_per_minute`
- `mitm_host` (combiné avec le JSON overlay)
- `broker.log_level` — déféré : `--log-level` n'est pas branché au backend swift-log (cf project memory limitation 5.3.1). Si on touche au log level ici, on doit le brancher partout — out of scope. Warn + ignore.

**Champs structurels non rechargeables** (changement → log warn, ignored) :
- `broker.listen` (port proxy)
- `broker.events_listen`
- `broker.admin_socket`
- `broker.event_retention_days`, `broker.event_ring_size`

**Comportement** :
- `Daemon.reload()` re-parse TOML depuis le `--config-path` original.
- Diff champ par champ. Pour chaque champ structurel modifié : `logger.warning("config.reload: ignoring change of <field> — restart required", metadata: ["old": ..., "new": ...])`. Surface non bloquante.
- Pour chaque champ hot-reloadable : applique via les hooks `ProxyServer.update*`.
- Si TOML invalide ou inaccessible : erreur JSON-RPC `-32012 configReloadFailed` avec underlying error message. Aucun champ n'est appliqué (transaction-like).

**SIGHUP** : `SignalHandling.installSIGHUP(handler:)` (nouveau, symétrique à `onSIGINTOnce` mais s'auto-réarme). Handler dispatch sur `Task { try? await daemon.reload() }`.

### 3.3 `events.clear` : vide le buffer, préserve les totaux cumulés

**Sémantique** : sur `EventRing` actor :
- `clear()` retire toutes les entrées du ring, retourne le nombre d'entrées supprimées.
- **Ne reset pas** `totals` (compteurs par `Event.Kind`). Cohérent avec project memory : « totaux par `Event.Kind` (jamais reset) pour `daemon.stats` ».

**Justification** : `daemon.stats` sert au menu-bar (Phase 6) qui affichera « N requêtes traitées depuis le démarrage du daemon ». Un `events.clear` qui reseterait les totaux ferait apparaître la vue avec compteurs à zéro alors que le daemon a effectivement traité du trafic. Désaligné de l'intent utilisateur (« vider l'historique consultable », pas « réinitialiser le daemon »).

### 3.4 `MITMRule.createdAt` pour les rules du TOML

Le TOML actuel a `[[mitm_host]] host = "api.openai.com"` sans timestamp. Pour `rule.list`, on synthétise un `createdAt` :
- **Choix** : `Date(timeIntervalSince1970: 0)` — sentinel « source: TOML ».
- Le CLI `iris rule list` distinguera dans l'affichage table : col `source` = `toml` ou `runtime`.

Alternative rejetée : mtime du fichier TOML (introduit du I/O à chaque list, non déterministe pour les tests).

## 4. Architecture des composants

### 4.1 Nouveau : `RuntimeRulesStore`

```swift
// Sources/IrisKit/Rules/RuntimeRulesStore.swift
public actor RuntimeRulesStore {
    public init(path: URL, logger: Logger) async throws
    public func list() -> [MITMRule]                     // sorted by host
    public func add(host: String, now: Date) throws -> MITMRule
    public func delete(host: String) throws -> Bool      // true if found+removed
    public func contains(host: String) -> Bool
}
```

- Init charge depuis disque (fichier absent = []).
- État interne : `[String: MITMRule]` keyed par host (dédup naturelle).
- Atomic write sur add/delete.
- Validation du host : DNS-like format, refusée si vide/contient `/`/contient espaces (réutilise pattern de `Config.validate` si existe, sinon regex simple `^[a-z0-9.-]+$`).

### 4.2 Modifié : `ProxyServer`

Ajoute 2 méthodes :
```swift
public func updateAllowedHosts(_ hosts: Set<String>) async
public func updateSecurityPolicy(
    maxSubstitutionsPerMinute: Int,
    onExfilAttempt: ExfilAttemptPolicy
) async
```

Storage interne : les champs `allowedHosts`, `maxSubstitutionsPerMinute`, `onExfilAttempt` deviennent `NIOLockedValueBox<T>` (déjà utilisé pour `isPaused` en Phase 3). `MITMHandler` lit la valeur à chaque requête (overhead négligeable). `ExfilRuleEngine` consomme aussi les deux derniers — il faut soit lui passer le getter (closure capturée), soit le rendre stateful sur ces fields. Préférer la closure : moins invasif.

### 4.3 Modifié : `Daemon`

- Stocke `configPath: URL` (passé au init pour permettre le reload).
- Nouveau : `public func reload() async throws -> ConfigReloadResult`
- Lit `configPath`, parse TOML (réutilise loader existant), diff vs `currentConfig` (var mutable maintenant), apply hot-reloadables, warn sur structural diffs.
- Stocke `runtimeRulesStore`, le passe au `AdminDispatcher` et hook `add/delete` → trigger `proxy.updateAllowedHosts(toml ∪ json)`.

### 4.4 Modifié : `AdminDispatcher`

- Nouveau collaborator : `runtimeRulesStore: RuntimeRulesStore`
- Nouveau collaborator : `reloadCallback: @Sendable () async throws -> ConfigReloadResult` (évite de pousser `Daemon` dans le dispatcher, garde le dispatcher `Sendable` sans contraintes lourdes).
- 5 nouveaux handlers : `rule.add`, `rule.list`, `rule.delete`, `config.reload`, `events.clear`.

### 4.5 Modifié : `EventRing`

Ajout method :
```swift
public func clear() -> Int {
    let n = entries.count
    entries.removeAll(keepingCapacity: true)
    return n
}
```

`totals` intouché par construction.

### 4.6 Modifié : `AdminProtocol`

Nouveaux cases :
```swift
case ruleAdd      = "rule.add"
case ruleList     = "rule.list"
case ruleDelete   = "rule.delete"
case configReload = "config.reload"
case eventsClear  = "events.clear"
```

Nouveaux types :
```swift
public struct RuleHostParams: Codable, Sendable, Equatable {
    public let host: String
}
public struct RuleListResult: Codable, Sendable, Equatable {
    public let rules: [MITMRule]      // tous (toml + runtime)
    // Note: MITMRule doit gagner un field `source` Codable, soit dans le wire
    // soit en computed côté CLI. Choix : wire-side. Voir §4.7.
}
public struct RuleDeletedResult: Codable, Sendable, Equatable {
    public let deleted: Bool
}
public struct ConfigReloadResult: Codable, Sendable, Equatable {
    public let reloaded: Bool
    public let ignored: [String]      // noms des champs structurels modifiés mais ignorés
}
public struct EventsClearResult: Codable, Sendable, Equatable {
    public let deletedCount: Int
    enum CodingKeys: String, CodingKey { case deletedCount = "deleted_count" }
}
```

Nouveaux codes d'erreur custom :
- `-32010 ruleProtected` — refus delete TOML rule
- `-32011 ruleNotFound`
- `-32012 configReloadFailed` — TOML parse error / I/O error

### 4.7 Modifié : `MITMRule`

Ajoute `source`:
```swift
public struct MITMRule: Codable, Sendable, Hashable {
    public enum Source: String, Codable, Sendable { case toml, runtime }
    public let host: String
    public let createdAt: Date
    public let source: Source
}
```

⚠️ **Risque migration** : ce field est `let` sur un type Codable. Tout consumer existant (memory du projet : Phase 6 menu-bar future) doit gérer. Default fallback côté Decoder pour les events historiques persistés : pas applicable (les `MITMRule` ne sont pas persistés aujourd'hui — uniquement dans la config TOML qui n'a pas ce field, et dans le wire JSON-RPC qui est éphémère).

**Solution sûre** : `source` reste obligatoire au wire (toujours présent en réponse). Les tests Phase 4 qui construisent des `MITMRule(host:, createdAt:)` doivent gagner `source: .toml`. Pas de breaking change persistant.

### 4.8 Modifié : `CLI iris`

- `Sources/iris/Commands/RuleCommands.swift` : `emitNotImplemented` remplacé par vraies impls qui consomment `rule.*` RPC.
  - `iris rule add <host>` → RPC `rule.add`, table 1 ligne ou JSON.
  - `iris rule list` → RPC `rule.list`, table `host | source | created_at`.
  - `iris rule rm <host>` → RPC `rule.delete`, gère `-32010` (exit 1 + message clair « rule defined in TOML, edit config file ») et `-32011` (exit 1 + « rule not found »).
- `Sources/iris/Commands/ConfigCommands.swift` : `config reload` débrayé → RPC `config.reload`. Affiche `ignored` si non vide.
- Pas de nouvelle commande `iris events clear` (pas dans SPECS §4.3). RPC accessible directement via le socket si besoin opérationnel.

## 7. Risques et mitigation

| Risque | Mitigation |
|---|---|
| `MITMRule.source` casse Phase 6 menu-bar future | Phase 6 pas commencée, on définit le contrat avant. Documenter dans memory. |
| Hot-swap des hosts MITM crée race avec requêtes en cours | `NIOLockedValueBox` snapshot read par requête. Race acceptable : la req en vol utilise l'ancien état, la suivante le nouveau. SPECS §13 ne garantit pas l'atomicité « toutes les reqs post-reload utilisent le nouveau set ». |
| TOML parse error pendant reload laisse daemon dans état incohérent | Reload transaction-like : parse + validate AVANT d'appliquer quoi que ce soit. Erreur `-32012` ne touche pas l'état actif. |
| `RuntimeRulesStore` write corrompu (disque plein) | Atomic write via `rename(2)` : soit ancien fichier intact, soit nouveau valide. Plus jamais d'état intermédiaire. |
| SIGHUP envoyé pendant un autre SIGHUP en cours | Handler s'arme indéfiniment. `Daemon.reload()` est `async` sur l'actor `Daemon` → sérialisé naturellement par Swift Concurrency. Deuxième SIGHUP attend le premier (acceptable). |

## 8. Out of scope (pour Phase 4.y / Phase 6 / Phase 8)

- SQLite events store (`events.query` historique persistant, retention timer).
- Validation forte du host (DNS lookup, etc.) — regex syntaxique suffit pour MVP.
- `--log-level` branché au backend swift-log.
- UI menu-bar pour gérer les rules (Phase 6).
- Rate limiting des appels admin RPC.
