# Phase 5.3.1 — JSONC support + `iris mcp wrap --watch`

**Date**: 2026-05-26
**Status**: design approuvé (brainstorming), à transformer en plan d'implémentation
**Phase précédente**: 5.3 (mcp wrap/unwrap one-shot, merged `8d50570`)
**Branche cible**: `feat/phase-5.3.1-jsonc-watch`

## 1. Objectif et portée

Étendre `iris mcp wrap` pour :
1. **Accepter les fichiers JSONC** (commentaires `//` `/* */` et virgules traînantes). Aujourd'hui le parser RFC 8259 strict refuse ces fichiers, ce qui force l'utilisateur à édulcorer son `~/.claude.json` avant d'invoquer la commande.
2. **Re-patcher automatiquement à chaque édition** via un flag `--watch`. Aujourd'hui chaque modification de `~/.claude.json` par Claude Desktop ou par l'utilisateur exige un `iris mcp wrap` manuel.

**Hors scope (différé) :**
- JSON5 (unquoted keys, single quotes, hex literals, `+`/`-Inf`/`NaN`).
- Multi-file watch dans un seul processus.
- Préservation des commentaires utilisateur lors du write-back.
- Output `--json` en mode `--watch` (NDJSON per-event).
- Notification daemon-side sur watch error (différée Phase 6 menu-bar).

## 2. Décisions de design

Trois décisions prises pendant brainstorming, à figer ici comme contraintes du plan.

### 2.1 — Commentaires : refuse-write, normalisation silencieuse des trailing commas

Le parser JSONC enregistre les positions des commentaires détectés. Si le document à patcher contient au moins un commentaire, `iris mcp wrap` refuse l'écriture (exit 65) avec un message qui localise le premier commentaire (`L<line>:<col>`). `--dry-run` reste autorisé sur un fichier commenté.

**Trailing commas** sont normalisées silencieusement (pas d'information perdue : `,]` et `,}` sont purement syntaxiques).

**Justification** : préserver les commentaires au round-trip exige ~300 lignes de tagging dans `OrderedJSONValue` + edge cases nombreux (commentaire entre `,` et clé suivante, doc-comments multi-lignes, bloc en fin de fichier). Stripper silencieusement viole CLAUDE.md §12 (fail loud). Refuser conserve l'invariant "aucune perte silencieuse" sans payer la complexité du round-trip JSONC.

### 2.2 — `--watch` est single-file par invocation

Forme CLI : `iris mcp wrap --watch <path>` watch un seul fichier. Pour 3 fichiers, 3 processus (composable via launchd ou un wrapper futur `iris watch run --config …`).

**Justification** : sémantique 1:1 avec `wrap` non-watch, code path simple (un seul `FSEventStream`, un seul hash, un seul debounce), faible surface de test. Multi-file est différable sans casser la sémantique actuelle.

### 2.3 — Résilience : keep watching, log loud

Trois failure modes pendant un cycle de watch, tous gérés par log warn + continuer (jamais d'exit) :
- **Daemon down** → retry exponentiel 1s→30s, ne pas exit.
- **Fichier disparu** → attendre l'event FSEvents de recréation.
- **JSON invalide en cours d'édition** → log warn une fois par cycle de debounce, attendre le prochain event.

Seuls **SIGINT** et **erreur fatale au démarrage** (path inexistant, daemon down au premier fetch) provoquent un exit.

**Justification** : un éditeur qui save de manière fugitive ne doit pas tuer un processus de watch long-running. La résilience est nécessaire à l'utilité opérationnelle.

### 3.1 — `OrderedJSONDocument` étendu

API publique additive :

```swift
extension OrderedJSONDocument {
    public struct ParseOptions: Sendable {
        public var mode: Mode
        public var recordCommentPositions: Bool
        public enum Mode: Sendable { case strict, jsonc }
        public static let strict = ParseOptions(mode: .strict, recordCommentPositions: false)
        public static let jsonc  = ParseOptions(mode: .jsonc,  recordCommentPositions: true)
    }

    public static func parse(_ input: String, options: ParseOptions = .strict)
        throws -> OrderedJSONDocument
}

public struct OrderedJSONDocument: Sendable {
    public private(set) var root: OrderedJSONValue
    public private(set) var commentPositions: [CommentPosition]   // NOUVEAU
}

public struct CommentPosition: Sendable, Equatable {
    public let line: Int        // 1-indexed
    public let column: Int      // 1-indexed
    public let kind: Kind
    public enum Kind: Sendable { case lineComment, blockComment }
}
```

**Backward-compat** : `parse(_:)` sans `options` reste strict, mêmes erreurs (`commentNotAllowed`, `trailingCommaNotAllowed`). Tests Phase 5.3 inchangés.

**Implémentation tokenizer** : un seul helper `skipWhitespaceAndComments()` qui branche selon le mode. ~50–70 lignes ajoutées au parser. Détails :
- En mode `.jsonc`, après `skipWhitespace()` standard, si `peek == "/"` :
  - `peek2 == "/"` → consomme jusqu'à `\n` ou EOF, enregistre la position si `recordCommentPositions`.
  - `peek2 == "*"` → consomme jusqu'à `*/`, throw `.unterminatedBlockComment(position:)` si EOF rencontré avant.
- Trailing commas : en mode `.jsonc`, après un élément d'array/object, si après skip on trouve `,` puis le terminateur (`]` ou `}`), accepter silencieusement.

**Nouvelles erreurs** :
- `.unterminatedBlockComment(position: Int)` — `/*` sans `*/` jusqu'à EOF.

**Serializer** : aucun changement. Émet du JSON pur. C'est `MCPCommand.Wrap` qui décide en amont si l'écriture est autorisée.

### 3.2 — `FileWatcher` (nouveau, dans `IrisKit`)

```swift
public final class FileWatcher: Sendable {
    public init(path: String, debounce: Duration = .milliseconds(500))
    public func events() -> AsyncStream<Void>
    public func stop()
}
```

**Implémentation** :
- `FSEventStreamCreate` sur le **parent directory** du path (les éditeurs font `write tmp + rename`, l'inode change, watcher fichier perdrait l'événement). Filtre interne sur `lastPathComponent` du path.
- Flags : `kFSEventStreamCreateFlagFileEvents | kFSEventStreamCreateFlagNoDefer`.
- Callback FSEvents schedulé sur une `DispatchQueue` dédiée label `io.iris.filewatcher.<uuid>`.
- Debounce : un `Task` réinitialisé à chaque event. Si un nouvel event arrive avant 500 ms, on annule la task pending et on en re-schedule une.
- `events()` retourne un `AsyncStream<Void>` qui émet `()` à chaque cycle débouncé.
- `stop()` invalide le FSEventStream et termine le stream.

**Placement dans `IrisKit`** plutôt que local à `iris` : code pur-Swift sans dépendance CLI, réutilisable (par exemple par un futur `iris watch run --config`). Décidé par user pendant brainstorming.

### 3.3 — Watch loop dans `MCPCommand.Wrap`

Pseudo-code :

```swift
func runWatch(path: String) async throws {
    let watcher = FileWatcher(path: path)
    var lastWrittenHash: Data? = nil
    var consecutiveFailures: Int = 0

    // Initial cycle (exit codes du démarrage applicables)
    try await runOneCycle(...)

    // SIGINT setup (réutilise SignalToken de Phase 5.2)
    let token = SignalHandling.onSIGINTOnce { watcher.stop() }
    defer { withExtendedLifetime(token) {} }

    // Merge file events + backoff retry ticks. AsyncStream.merge n'existe pas
    // dans la std lib Swift — soit ajouter swift-async-algorithms (officiel
    // Apple) pour `merge(_:_:)`, soit écrire un merge manuel à base d'une
    // AsyncStream factory qui spawne 2 Tasks de relais. À trancher au plan.
    let merged = mergeEvents(watcher.events(), backoffTicks)
    for await _ in merged {
        do {
            try await runOneCycle(...)
            consecutiveFailures = 0
            cancelBackoffTimer()
        } catch let error as DaemonUnreachable {
            consecutiveFailures += 1
            log.warning("daemon unreachable, retrying in \(delay(consecutiveFailures))s")
            armBackoffTimer(delay(consecutiveFailures))
        } catch {
            log.warning("cycle failed: \(error)")
        }
    }
}
```

`runOneCycle` :
1. `Data(contentsOf: fileURL)` — si disparu, log info, return.
2. `SHA256(data)` — si `== lastWrittenHash`, return (notre propre write).
3. Fetch daemon config (broker, ca) — peut throw `DaemonUnreachable`.
4. `OrderedJSONDocument.parse(text, options: .jsonc)` — si throw, log warn, return.
5. Si `commentPositions.isEmpty == false`, log warn `L<line>:<col>`, return.
6. `MCPPatcher.patch(...)` → `(patched, summary)`.
7. `serialize(patched)` → bytes.
8. Si `bytes == data`, déjà compliant, `lastWrittenHash = SHA256(bytes)`, return.
9. Backup atomique (`.iris.bak`) → write atomique → `lastWrittenHash = SHA256(bytes)` → log info.

## 4. Data flow d'un cycle

```
FSEventStream callback
    │
    ▼
debounce 500 ms (cancel pending, schedule new)
    │
    ▼
events() emit ()
    │
    ▼
runOneCycle:
    read file ─── disparu ─→ log info, return
    │
    SHA256 == lastWrittenHash ? ─── yes ─→ return (notre write)
    │ no
    fetch daemon config ─── DaemonUnreachable ─→ log warn, arm backoff, return
    │
    parse JSONC ─── fail ─→ log warn, return
    │
    commentPositions empty ? ─── no ─→ log warn L:C, return
    │ yes
    patch ─── serialize ─── bytes == original ? ─── yes ─→ update hash, return
    │                                              │ no
    backup atomique → write atomique
    │
    lastWrittenHash = SHA256(patched bytes)
    │
    log info "patched"
```

**Race "user édite pendant notre write"** : on écrit à T=0. FSEvents notifie à T+ε. Debounce 500 ms. Au cycle T+500 ms, on rehash. Si user a sauvegardé entre temps avec contenu différent, hash diff → on re-processe. Pas de fenêtre de perte d'edit.

**Initial cycle** : avant la boucle, on appelle `runOneCycle` une fois. Si path inaccessible → exit 65. Si daemon down → exit 2 (cohérent avec les autres sous-commandes `iris`). Le backoff ne s'arme **qu'après** le premier fetch réussi.

**`--watch` + `--dry-run`** : conflit sémantique, ArgumentParser refuse avec exit 64.

**`--watch` + `--json`** : différé. En mode `--watch`, output human-readable sur stderr via `swift-log`.

**SIGINT** : `SignalToken` (Phase 5.2), stop FSEventStream, exit 0.

## 5. Erreurs et exit codes

### 5.1 — Démarrage

| Condition | Exit | Notes |
|---|---|---|
| `--watch` + `--dry-run` | 64 | ArgumentParser |
| Path inaccessible / parent dir unreadable | 65 | stderr + errno |
| Daemon down au premier fetch | 2 | `withAdminClient` standard |
| Path est `.iris.bak` | 65 | régression Phase 5.3 |

### 5.2 — Pendant un cycle (jamais d'exit, log loud)

| Condition | Niveau | Action |
|---|---|---|
| Fichier disparu | `info` | wait next event |
| Read I/O error | `warn` | wait next event |
| Hash match | `debug` | skip silencieux |
| Parse fail JSONC | `warn` (1×/cycle) | wait next event |
| Commentaires détectés | `warn` (1×/cycle, `L:C`) | wait next event |
| Daemon down mid-watch | `warn` (1×/burst) | arm backoff |
| Substitution OK, write I/O error | `error` | wait next event |
| File already compliant | `debug` | success, update hash |
| Write réussit | `info` | success, update hash |

### 5.3 — Backoff timer

État : `consecutiveFailures: Int`. Reset à 0 sur succès.

```
delay(n) = min(2^(n-1), 30) seconds   pour n ≥ 1
         = 1, 2, 4, 8, 16, 30, 30, …
```

Events FSEvents en compétition avec le timer : un edit utilisateur pendant un backoff window déclenche un cycle immédiat sans réinitialiser le compteur (succès → reset ; échec → `n+1`).

### 5.4 — Format de log

Sortie stderr, human-readable, une ligne par event :

```
2026-05-26T18:42:13Z [info]  iris.watch: started, watching /Users/.../.claude.json
2026-05-26T18:42:13Z [info]  iris.watch: file already compliant
2026-05-26T18:45:01Z [warn]  iris.watch: comments detected at L8:3 — refusing to write
2026-05-26T18:47:22Z [info]  iris.watch: patched (backup at .claude.json.iris.bak)
2026-05-26T18:50:00Z [warn]  iris.watch: daemon unreachable, retrying in 1s
2026-05-26T18:50:08Z [info]  iris.watch: daemon back, resuming
2026-05-26T18:55:00Z [info]  iris.watch: stopped (SIGINT)
```

Niveau par défaut `info`. Flag `--log-level <debug|info|warn|error>` ajouté à `wrap`, utile seulement en mode `--watch`.

**Invariant log (test locké)** : aucune valeur de secret n'apparaît dans les logs. Tenu par construction (le wrap ne substitue pas — il ajoute des env vars proxy + CA path ; les valeurs secrets ne transitent que par le daemon à runtime).

## 8. Écarts SPECS et points à valider

- **`OrderedJSONDocument.Equatable`** : avec `commentPositions` désormais dans le doc, `==` les compare. Si on veut comparer le "contenu logique" (idempotence patcher), comparer `.root` directement. À documenter dans le doc-comment du type.
- **Tests FS-flaky** : convention `getenv("SKIP_FS_TESTS")` à introduire si elle n'existe pas déjà (à vérifier dans le harness). Sinon tests skipés conditionnellement via une autre voie.
- **`swift-log` backend** : déjà dans `Package.swift` (utilisé par `irisd`). Vérifier que le CLI peut en bénéficier sans setup particulier — `LoggingSystem.bootstrap` à appeler en début de `runWatch`. Format ISO 8601 timestamp + label `iris.watch`.
- **AsyncStream merge** : pas de `merge` natif std lib. Option A — ajouter `swift-async-algorithms` (officiel Apple, ferait sens projet-wide). Option B — petit helper local qui spawne 2 Tasks de relais. À trancher au plan d'implémentation.
- **`FileWatcher` : `actor` ou `final class`** ? Le state mutable (FSEventStream ref, debounce task) est manipulé depuis le callback FSEvents (queue dédiée) et depuis `stop()` (appelant). `actor` est plus sûr mais le callback FSEvents est sync C, donc on aura besoin d'un `nonisolated(unsafe)` ou d'un détour `Task.detached`. À trancher au plan.
- **`--log-level` flag** : ajouté à la sous-commande `wrap`, **utile uniquement en mode `--watch`**. En mode one-shot, output reste sur stdout/stderr courant via `print` / `FileHandle.standardError.write` (cohérence Phase 5.3). Le flag est accepté mais sans effet en one-shot ; un warning peut être émis si besoin.
- **Aucun écart SPECS connu introduit par cette phase.** Pas de modification de l'API admin RPC, pas de modification du daemon, pas d'impact sur le proxy MITM.
