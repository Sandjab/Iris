# Phase 6.2.x — Quarantine secret

> Désactivation temporaire de la substitution d'un secret, persistée comme
> propriété du secret en Keychain, pilotable depuis le CLI et l'app menu-bar.
> Source SPECS : §15.2.3 (« Quarantine secret » action, onglet Security).

## 1. Objectif et portée

Permettre de **neutraliser un secret** sans le supprimer ni le faire tourner :
tant qu'il est en quarantaine, ses placeholders `{{kc:NAME}}` ne sont jamais
substitués et le secret est totalement inerte vis-à-vis du moteur
d'exfiltration. L'opération est réversible (sortie de quarantaine), persistée au
reboot, et son état est visible partout (CLI, app).

**Dans le périmètre :**
- Champ `quarantined` (défaut `false`) sur le modèle `Secret`, persisté en Keychain.
- Nouvelle méthode `SecretStore.setQuarantined`, implémentée Keychain + InMemory.
- RPC daemon `secret.set_quarantined`.
- Skip dans le hot path de substitution (`ExfilRuleEngine.evaluate`).
- CLI `iris secret quarantine` / `unquarantine` + affichage de l'état.
- App : toggle dans l'onglet Secrets + action « Quarantine secret » dans l'onglet Security ; rendu visuel différencié.

**Hors périmètre (différé) :**
- ACL Keychain signed-binary (Phase 8, déjà documenté).
- Quarantine par `(secret, host)` granulaire — ici c'est tout-ou-rien par secret.
- Expiration automatique de la quarantaine (TTL) — réactivation manuelle uniquement.

## 2. Décisions de design

### 2.1 — Persistance : propriété du secret en Keychain (pas d'overlay)

Le flag vit dans le record Keychain du secret, dans le blob JSON `StoredMetadata`
(`kSecAttrGeneric`), à côté de `allowed_hosts`/timestamps/usage. Source de vérité
unique, survit au reboot, cohérent avec « until manually re-enabled » (SPECS §15.2.3).

**Justification** : l'alternative (overlay `quarantine.json` daemon-side) ajoutait
une persistance parallèle et un risque d'entrées orphelines. Le blob étant déjà
du JSON, ajouter un champ est trivial et sans migration. Choix utilisateur explicite.

**Pré-requis débloqué** : le dossier source `Sources/IrisKit/Secrets/` a été renommé
`SecretStore/` (commit `chore(phase-6.2.x)`) pour lever la collision avec le deny
global `Read(**/secrets/**)`. Aucun secret réel n'est dans ce dossier (code seul).

### 2.2 — Sémantique hot path : inerte total

Un secret en quarantaine est traité comme **absent** du pipeline : ni substitué,
ni évalué par les règles d'exfiltration.

| Cas | Secret normal | Secret en quarantaine |
|---|---|---|
| host autorisé | substitué | laissé littéral |
| host interdit | bloqué (R1 hostMismatch) | inerte (pas de block, littéral) |
| co-présent avec un secret actif | peut déclencher R3 | exclu de R3 ; l'actif est traité seul |

**Justification** : un secret en quarantaine ne substitue aucune valeur réelle →
aucune fuite possible → faire bloquer du trafic par un secret inerte serait
surprenant et sans bénéfice sécurité. L'alternative « alerter sans substituer » a
été écartée (verdict utilisateur). Les placeholders inconnus (typos) restent comptés
par R3, comportement inchangé.

### 2.3 — RPC : setter idempotent (pas une paire)

Un seul `secret.set_quarantined(name, quarantined: Bool)` côté daemon, idempotent.
La distinction quarantine/unquarantine est portée par le booléen. Le CLI expose en
revanche **deux** sous-commandes (`quarantine`/`unquarantine`) pour la découvrabilité,
qui appellent toutes deux le même RPC.

### 2.4 — Préservation du flag aux mutations

`update` (édition allowed_hosts) et `recordUsage` reconstruisent le blob
`StoredMetadata` depuis l'état courant → ils **doivent recopier `quarantined`**,
sinon une édition de hosts réactiverait silencieusement un secret en quarantaine.
`rotate` ne touche que `kSecValueData` → préserve déjà. Invariant verrouillé par test.

### 2.5 — UI : les deux onglets

- **Secrets tab** : icône toggle quarantaine à côté de edit/rotate/delete ; un secret
  en quarantaine est rendu **rouge + italique**.
- **Security tab** : action « Quarantine secret » sur un event exfil → met en
  quarantaine le secret nommé par l'alerte (SPECS §15.2.3).

Les deux passent par la même mutation `AppModel` (RPC → re-fetch, pattern 6.2).

## 3. Architecture

### 3.1 Modules touchés

```
Sources/IrisKit/
  Models/Secret.swift                      (+ champ quarantined)
  SecretStore/SecretStore.swift            (+ setQuarantined au protocole)
  SecretStore/KeychainSecretStore.swift    (+ champ StoredMetadata, + setQuarantined, préservation)
  SecretStore/InMemorySecretStore.swift    (+ setQuarantined)
  IPC/AdminProtocol.swift                  (+ case + params)
  IPC/AdminDispatcher.swift                (+ handler)
  Placeholder/ExfilRuleEngine.swift        (skip inerte)
Sources/iris/Commands/SecretCommands.swift (+ quarantine/unquarantine, affichage état)
Sources/IrisAppCore/                       (AdminCalling + AppModel : setQuarantined)
IrisApp/IrisApp/                           (SecretsTab toggle + rendu, SecurityTab action)
```

### 3.2 Couche store

```swift
// SecretStore (protocole)
func setQuarantined(_ quarantined: Bool, named name: String) async throws -> Secret
```
- **Keychain** : lire le secret courant, réécrire le blob avec `quarantined` modifié
  (réutilise le chemin `updateMetadata`), renvoyer le `Secret` à jour. `unknownSecret`
  si absent.
- **InMemory** : muter `entry.metadata`, renvoyer le `Secret` à jour.

`StoredMetadata` gagne `let quarantined: Bool` avec `decodeIfPresent(...) ?? false`
au décodage (rétro-compat des records écrits avant cette phase).

### 3.3 Hot path

Dans `ExfilRuleEngine.evaluate`, après le fetch des métadonnées par hit (déjà
existant, ligne ~99) :

```swift
let effectiveHits = hits.filter { metadataByName[$0.name]?.quarantined != true }
```

Puis dérouler R1-R5 et construire `knownHits`/`resolvable` à partir de `effectiveHits`
au lieu de `hits`. Les inconnus (`metadataByName[name] == nil`) restent dans
`effectiveHits` (R3 typo detection préservée). **Zéro lecture Keychain
supplémentaire** : le flag voyage sur le `secret(named:)` déjà effectué.

### 3.4 RPC + IPC

- `AdminMethod.secretSetQuarantined = "secret.set_quarantined"`.
- `SecretQuarantineParams { name: String, quarantined: Bool }` (CodingKeys snake_case explicites, cf décision wire-format Phase 3).
- Résultat : `Secret` (qui porte désormais `quarantined`).
- `secret.list` / `secret.get` remontent le flag sans changement (ils renvoient déjà `Secret`).
- `DaemonVersion.current` → `"0.6.0-phase6.2.x"`.

### 3.5 App (IrisAppCore + IrisApp)

- `AdminCalling` : `func setQuarantined(name:quarantined:) async throws -> Secret`, conformance IrisKit.
- `AppModel` : action `setQuarantined(name:quarantined:)` → appel RPC → re-fetch `secret.list` (pattern 6.2, pas d'optimistic). Conserve le bandeau d'erreur via `RPCError.userMessage`.
- `SecretsTab` : colonne d'actions + icône toggle (état dérivé de `secret.quarantined`) ; modificateurs `.foregroundStyle(.red)` + `.italic()` conditionnels.
- `SecurityTab` : bouton « Quarantine secret » par alerte → `setQuarantined(name: alert.secretName, quarantined: true)`.

### 3.6 Flow données

```
[Secrets tab toggle] / [Security action]
        │ AppModel.setQuarantined(name, q)
        ▼
AdminClient.call(secret.set_quarantined)
        ▼ daemon
AdminDispatcher → store.setQuarantined → Keychain blob réécrit
        ▼ réponse Secret
AppModel re-fetch secret.list → @Published secrets → UI rerender
                                        │
        (requêtes suivantes) ExfilRuleEngine lit quarantined → skip inerte
```

## 4. Edge cases et error handling

| Cas | Comportement |
|---|---|
| Record Keychain pré-existant sans champ | `decodeIfPresent ?? false` → non quarantiné. |
| `setQuarantined` sur secret inexistant | `unknownSecret` → RPC `-32602`/message, CLI exit non-zéro, app bandeau. |
| Édition allowed_hosts d'un secret en quarantaine | `update` préserve `quarantined` (invariant §2.4). |
| `rotate` d'un secret en quarantaine | flag préservé (ne touche que la valeur). |
| Quarantine pendant que la valeur est en cache (`PlaceholderEngine` TTL 5 min) | sans effet : le skip se fait en amont dans `evaluate`, le cache de valeur n'est jamais consulté pour un nom quarantiné. |
| Daemon down (app) | erreur connexion → bandeau, pas de mutation locale. |
| Quarantine + sortie rapide (idempotence) | setter idempotent, dernier état gagne après re-fetch. |

## 5. Sync : rafraîchissement de l'état

Mutation = RPC puis re-fetch `secret.list` (cohérent avec 6.2). Aucune diffusion SSE
dédiée : l'état quarantine n'est pas un event de trafic. L'app reflète le flag au
re-fetch post-mutation et au bootstrap.

## 6. Testing strategy

### 6.1 Tests automatisés (`IrisKitTests` + `IrisAppCoreTests`)

Hot path (CLAUDE.md §6.3 — régression scoping, locké) :
- Secret quarantiné + host autorisé → **non substitué**, placeholder littéral.
- Secret quarantiné + host interdit → **pas de block** (inerte), littéral.
- Secret quarantiné co-présent avec un actif → actif substitué, quarantiné littéral, R3 non déclenché par la paire.
- Sortie de quarantaine → substitution reprend.
- Régression : un secret non-quarantiné conserve exactement R1-R5 (aucune dérive).

Store (via `InMemorySecretStore` — CLAUDE.md §7 : pas de test sur vrai trousseau, abstraction derrière le protocole) :
- `setQuarantined` aller/retour.
- `update`/`recordUsage` **préservent** `quarantined` (invariant §2.4).
- `rotate` préserve `quarantined`.
- Rétro-compat : décodage d'un `Secret` (Codable public) sans le champ → `false`. Le blob `StoredMetadata` du `KeychainSecretStore` (privé, non unit-testable sans trousseau) applique le même `decodeIfPresent ?? false`.

RPC + App :
- `secret.set_quarantined` round-trip via `AdminDispatcher`.
- `AppModel.setQuarantined` → re-fetch, `@Published` à jour.
- Invariant no-secret-leak maintenu (aucune valeur dans les events/alertes — déjà locké, vérifier non-régression).

### 6.2 CI

Aucun nouveau job. `swift build` + `swift test` (macos-14) + `xcode-build` (macos-15,
hard gate, depuis 6.2) + `swift-format lint --strict --recursive Sources Tests Package.swift IrisApp`.

### 6.3 Smoke checklist PR (cases à cocher avant merge)

- [ ] `iris secret quarantine <name>` → `iris secret show <name>` indique l'état quarantiné.
- [ ] Trafic via proxy avec `{{kc:<name>}}` vers host autorisé : **pas** de substitution tant que quarantiné (placeholder littéral upstream).
- [ ] `iris secret unquarantine <name>` → substitution reprend.
- [ ] App Secrets tab : icône toggle, secret en quarantaine rendu rouge + italique.
- [ ] App Security tab : action « Quarantine secret » sur une alerte met le secret en quarantaine (visible dans Secrets tab après re-fetch).
- [ ] Édition des allowed_hosts d'un secret en quarantaine ne le réactive pas.
- [ ] Aucune valeur de secret dans logs/events/alertes (invariant §6.1).

## 8. Limitations connues / différé explicite

- Pas de TTL d'auto-réactivation : réactivation manuelle uniquement.
- Pas de quarantine granulaire `(secret, host)` : tout-ou-rien par secret.
- ACL Keychain signed-binary toujours différée Phase 8 (orthogonal).
- L'overlay `config.get` boot-time post-reload reste différé (sans impact ici).
