# Phase 6.3a — Modèle de configuration unifié (app-first, fichier unique)

> Design validé le 2026-06-05. Prérequis backend de la Phase 6.3 (onglet Settings
> menu-bar, SPECS §15.2-6). La 6.3 a été scindée en **6.3a** (cette spec : refonte
> du modèle de config) et **6.3b** (UI Settings, qui consomme 6.3a — capturée en
> §11). Issue d'un brainstorming qui, parti de « read-only vs éditable », a
> convergé vers un changement de **propriété** de la configuration.

## 1. Objectif & motivation

Faire passer la configuration d'IRIS d'un modèle **« l'humain possède le fichier »**
(TOML déclaratif édité à la main, lu par le daemon) à un modèle **app-first
« la machine possède le fichier »** : un **fichier JSON unique**, écrit
exclusivement par le daemon, **seedé** à la première exécution et **enrichi via
l'app/CLI**. Objectif produit : l'édition des réglages **sans friction** depuis
l'UI (G7 SPECS — no-friction), sans imposer l'ouverture d'un éditeur de texte.

Le modèle « base déclarative + couche runtime persistée à part » existe **déjà**
pour les hosts MITM : `RuntimeRulesStore` (`Sources/IrisKit/Rules/RuntimeRulesStore.swift`)
persiste les hosts ajoutés à chaud dans `runtime-rules.json`, et `Daemon.reload()`
recalcule `allowedHosts = hosts TOML ∪ hosts runtime` (`Daemon.swift:337-340`).
Cette refonte **généralise** ce pattern à toute la config et **fusionne les deux
fichiers en un seul**.

**Non-objectif :** déplacer quoi que ce soit hors du Keychain. Les valeurs de
secrets et la clé privée CA restent dans le Keychain (CLAUDE.md §6.5, invariant
8b) ; ce fichier ne contient **aucun** secret.

## 2. Décisions de cadrage (verrouillées)

| Décision | Choix | Justification |
|----------|-------|---------------|
| Propriétaire du fichier | **Le daemon, seul écrivain** | L'app/CLI ne touchent jamais le fichier ; elles passent par RPC. Préserve le modèle de sécurité actuel (0600, write atomique, ACL) et évite le problème « deux auteurs, un fichier ». |
| Format | **JSON** | Machine-écrit (commentaires sans objet). Déjà le format de `runtime-rules.json`. **TOML abandonné** → dépendance `TOMLKit` retirée. |
| Périmètre du fichier | **Réglages + hosts MITM fusionnés** | `config.toml` **et** `runtime-rules.json` disparaissent au profit d'un seul `config.json`. |
| Champ `origin` des hosts | **Gardé + protection** | Remplace `MITMRule.source` (`.toml`/`.runtime`) par `origin` (`default`/`user`). Un host `origin: default` (seedé) est **non-supprimable** via RPC ; les `user` (ajoutés) restent librement supprimables. Évite de casser `claude` en retirant `api.anthropic.com` par mégarde. Décision utilisateur 2026-06-05. |
| Migration | **Aucune (greenfield)** | Aucun `config.toml` n'est déployé (absent du poste de dev, projet pré-1.0 non distribué). Pas de code de migration legacy. |
| Seed | **Au 1er démarrage du daemon** si fichier absent | Plus robuste que de dépendre du postinstall `.pkg` ; fonctionne aussi en dev/CLI. Source = `Config.default` (`Config.swift:22-36`). |
| `log_level` à chaud | **Hors-scope** | Le câblage du log level au backend logging est un TODO connu (`Daemon.swift:320-325`, « not yet wired to backend (Phase 5.3.1.x) »). Traité comme restart-required. |
| API wire RPC | **Stable** | `config.get`, `config.reload`, `rule.add/list/delete` gardent leur signature ; seul leur backend de stockage change. Ajout d'un seul nouveau verbe : `config.set`. |
| `version` de schéma | **`version: 1`** dans le fichier | Coût nul, dé-risque l'évolution future du schéma. |

## 3. Architecture : l'acteur `ConfigStore`

Nouveau `public actor ConfigStore` dans `IrisKit` (emplacement proposé :
`Sources/IrisKit/Config/ConfigStore.swift`). Il **absorbe** les responsabilités
aujourd'hui éclatées entre `ConfigLoader` (parse TOML) et `RuntimeRulesStore`
(persistance JSON des hosts), qui sont supprimés/fusionnés.

```
        App  ──RPC──┐
                    ├──►  ConfigStore (actor)  ──►  config.json   (0600, tmp→replace atomique)
        CLI  ──RPC──┘          │                        └─ backup horodaté avant chaque save
                               ▼
                  applique à chaud au daemon en cours :
                  proxy.updateSecurityPolicy(...)  /  proxy.updateAllowedHosts(...)
```

Responsabilités du `ConfigStore` :
- **Possède** l'unique `config.json` ; sérialise toutes les lectures/écritures (isolation acteur).
- **Charge** au boot ; **seed** si absent (§6).
- **Valide** (`Config.validate()`, `Config.swift:120-127`) avant tout persist ; un set invalide est **rejeté** (erreur RPC, fichier inchangé).
- **Persiste** en write atomique 0600 (réutilise le pattern `RuntimeRulesStore.persist()`, `RuntimeRulesStore.swift:104-137` : tmp + `setAttributes` 0600 + `replaceItemAt`).
- **Backup** avant chaque persist (§7).
- **Snapshot** courant pour `config.get` et pour le calcul de `allowedHosts`.

Le `Daemon` (`Sources/irisd/Daemon.swift`) consomme le `ConfigStore` au lieu de
`ConfigLoader` + `RuntimeRulesStore`. `Daemon.reload()` (`Daemon.swift:254`)
devient « relire le fichier depuis le disque et ré-appliquer » — utile pour le cas
où le fichier a été édité à la main, et reste branché sur SIGHUP (`App.swift:101-105`).

## 4. Schéma `config.json`

```json
{
  "version": 1,
  "broker": {
    "listen": "127.0.0.1:8888",
    "events_listen": "127.0.0.1:8899",
    "admin_socket": "~/Library/Application Support/iris/admin.sock",
    "log_level": "info",
    "event_retention_days": 7,
    "event_ring_size": 10000
  },
  "security": {
    "on_exfil_attempt": "block_and_notify",
    "max_substitutions_per_minute": 60
  },
  "backups": {
    "max_count": 10
  },
  "hosts": [
    { "host": "api.anthropic.com", "origin": "default", "created_at": "2026-06-05T12:00:00Z" }
  ]
}
```

- `broker` / `security` : champs et clés snake_case **identiques** à l'actuel
  `BrokerConfig` / `SecurityConfig` (`Config.swift:39-103`). Pas de changement de
  forme côté `Config`.
- `backups.max_count` : entier ≥ 1, **défaut 10**. Lu par `ConfigStore` à chaque
  save (donc modifiable à chaud, jamais besoin de restart).
- `hosts` : remplace `mitm_host` (TOML) **et** `runtime-rules.json`. Chaque entrée
  = `{ host, origin, created_at }`. `origin` ∈ `default` | `user` (voir §2 :
  protection des `default`). `created_at` conservé car l'onglet Rules affiche une
  colonne « added » (SPECS §15.2-5). Pour le seed, `created_at` = instant du seed,
  `origin: default`.

Le modèle interne `Config` est étendu pour porter `version`, `backups` et la forme
`hosts` (host + origin + created_at) en remplacement de `mitmHosts: [MITMHostEntry]`.
`MITMHostEntry` (host seul, `Config.swift:111-117`) et le `source` de `MITMRule`
(`.toml`/`.runtime`) fusionnent dans ce concept unique, le `source` devenant
`origin` (`.default`/`.user`).

## 5. API RPC & application

L'API au fil reste stable ; seul le backend change. Un seul verbe ajouté.

| RPC | Statut | Effet |
|-----|--------|-------|
| `config.get` | existant | Retourne le snapshot `Config` du store. |
| `config.set` | **nouveau** | Applique un patch de réglages scalaires. Valide → applique à chaud si possible → persiste (+backup) → retourne `{ applied: [...], requires_restart: [...] }`. |
| `rule.add` / `rule.delete` / `rule.list` | existants | Inchangés au fil ; écrivent/lisent désormais `config.json` via `ConfigStore` au lieu de `runtime-rules.json`. |
| `config.reload` | existant | Relit `config.json` depuis le disque et ré-applique (cas édition manuelle). |

**Granularité de `config.set`** : une liste d'updates `[{ key, value }]` où `key`
est un dot-path d'un **ensemble fermé** de clés autorisées
(`security.on_exfil_attempt`, `security.max_substitutions_per_minute`,
`broker.log_level`, `broker.listen`, `broker.events_listen`,
`broker.admin_socket`, `broker.event_retention_days`, `broker.event_ring_size`,
`backups.max_count`). Toute clé inconnue est rejetée. Le daemon construit le
`Config` candidat, le valide globalement, puis applique. Les **hosts ne passent
pas par `config.set`** (ils ont leur validation + `created_at` propres) : ils
restent gérés par `rule.add`/`rule.delete`.

**Protection des hosts `default`** : `rule.delete` sur un host `origin: default`
est **rejeté** par une erreur RPC dédiée (`JSONRPCError.hostProtected(host:)`) ;
`rule.add` crée un host `origin: user` ; `rule.list` renvoie `origin`. Pas de flag
`--force` (assumé : pour un setup volontairement sans `api.anthropic.com`, éditer
l'`origin` du host à la main).

**Hot-swap vs restart** (reflète le comportement runtime réel, `Daemon.swift:328-340`) :

| Réglage | Application |
|---------|-------------|
| `security.on_exfil_attempt` | ✅ à chaud (`proxy.updateSecurityPolicy`) |
| `security.max_substitutions_per_minute` | ✅ à chaud (`proxy.updateSecurityPolicy`) |
| `hosts` (via `rule.*`) | ✅ à chaud (`proxy.updateAllowedHosts`) |
| `backups.max_count` | ✅ à chaud (relu à chaque save) |
| `broker.listen` / `events_listen` / `admin_socket` | 🔁 restart requis (re-bind) |
| `broker.event_retention_days` / `event_ring_size` | 🔁 restart requis |
| `broker.log_level` | 🔁 restart requis (backend non câblé — hors-scope) |

Pour un réglage restart-required, `config.set` **persiste quand même** la valeur et
la renvoie dans `requires_restart` afin que l'UI (6.3b) puisse l'afficher
honnêtement (« prendra effet au redémarrage »).

CLI miroir : `iris config set <key> <value>` (une clé par appel) ; `iris config
get` et `iris config reload` inchangés (`Sources/iris/Commands/ConfigCommands.swift`).

## 6. Seed à la première exécution

Au démarrage, si `config.json` est absent à l'emplacement résolu, le daemon écrit
le contenu de `Config.default` (`Config.swift:22-36`) sérialisé en JSON, avec
`version: 1`, `backups.max_count: 10`, et `hosts: [{ host: "api.anthropic.com",
origin: "default", created_at: <now> }]`. Le seed est un write atomique 0600 (mêmes garanties que les
autres écritures). Aucun backup n'est créé pour le seed (il n'y a rien à
sauvegarder).

## 7. Backups

Avant **chaque** persist (hors seed) :
1. Lire l'état actuel sur disque.
2. Le copier vers `backups/config-<ISO8601>.json` (sous-dossier du répertoire de
   support, créé si besoin, 0600).
3. Écrire le nouveau contenu (atomique).
4. **Rotation** : si le nombre de backups dépasse `backups.max_count`, supprimer
   les plus anciens (tri par horodatage du nom) jusqu'à revenir au cap.

`max_count` est relu depuis le `Config` courant à chaque save. Filet de sécurité
contre une mauvaise écriture ou une édition manuelle écrasée par un `config.set`
ultérieur (risque inhérent au modèle « la machine possède le fichier »).

## 8. Rupture & impact

- **Fichiers supprimés** : `config.toml` (concept), `runtime-rules.json` (concept).
  `ConfigLoader` (parse TOML) et `RuntimeRulesStore` sont retirés/fusionnés dans
  `ConfigStore`.
- **Dépendance `TOMLKit` retirée** de `Package.swift` (plus aucun usage TOML).
- **`Daemon`** : `init` et `reload()` consomment `ConfigStore`. Le merge
  `tomlHosts ∪ runtimeHosts` (`Daemon.swift:100-103, 337-340`) devient « hosts du
  store ».
- **`App.swift`** : le flag `--config` et les `defaultPath` passent de
  `config.toml` à `config.json` (`App.swift:86, 117`). L'aide est mise à jour
  (`App.swift:20`).
- **`AdminProtocol`** : ajout du verbe `config.set` + ses types de payload/résultat ;
  **renommage `MITMRule.source` (`.toml`/`.runtime`) → `MITMRule.origin`
  (`.default`/`.user`)** (option « propre » retenue 2026-06-05).
- **`AdminDispatcher`** : route `config.set` vers `ConfigStore` ; `rule.*` repointés
  vers `ConfigStore` ; `rule.delete` rejette un host `origin: default`
  (`hostProtected`).
- **App + CLI adaptés au renommage** (sort la 6.3a du « backend-only ») :
  - `IrisApp/IrisApp/RulesTab.swift` : le cadenas/non-suppression, aujourd'hui
    conditionné par `source == .toml`, passe à `origin == .default`.
  - `Sources/iris/Commands/RuleCommands.swift` : `rule list` affiche `origin`
    (`default`/`user`) au lieu de `source`.
  - ⚠️ **Conséquence CI** : la 6.3a fait désormais compiler l'app → le **gate
    xcodebuild macos-15** s'applique. Le build local (toolchain en avance) n'est
    **pas** l'oracle de ce gate : pousser et vérifier le CI avant merge.
- **SPECS.md amendé** : §15.2-6 (Settings : « édition via config.toml » → édition
  in-app), la section config (TOML → JSON app-first), l'arborescence (§A
  `config.toml` → `config.json` + `backups/`), retrait des mentions TOML.
- **`docs/user-guide.md`** : section config resynchronisée (TOML → JSON), tableau
  d'état.

## 9. Gestion d'erreur

- **Fichier illisible / JSON corrompu au boot → boot dégradé, pas refus de boot.**
  Raison (décision utilisateur 2026-06-05) : un daemon qui `exit(1)` perd ses
  propres canaux (socket admin + SSE), donc l'app ne voit que
  `rpcError`/`notRunning` *sans la raison*, et sous LaunchAgent (Phase 7)
  `launchd` le relance en boucle (throttle 30 s, SPECS §A.5). Refuser de booter
  est en réalité le choix le **moins** audible côté utilisateur. Séquence retenue :
  1. **Backup** du fichier corrompu (ne pas écraser une config peut-être
     récupérable à la main).
  2. **Ré-seed** sur `Config.default` et **démarrage normal** : le daemon reste
     joignable, la **substitution reste active** (défauts sains — host =
     `api.anthropic.com` seul, `on_exfil_attempt: block_and_notify` — donc plutôt
     *moins* permissif que permissif).
  3. **Émettre une `Alert` de démarrage `severity: .high`** via le canal Event
     existant (`Event`/`Alert` → ring → SSE, déjà utilisé par les exfils :
     `ExfilRuleEngine`) → visible dans l'onglet Security, en notification, et via
     `iris doctor`/`iris logs`. C'est le « fail loud » qui atteint réellement
     l'utilisateur, puisqu'il exige que le daemon soit up. *Nouveau chemin* : le
     mécanisme `Alert` existe mais n'est aujourd'hui invoqué que par le pipeline
     de requête ; vérifier que `Event` (`Event.swift`) accepte une alerte sans
     contexte de requête, sinon l'adapter (champ requête optionnel). Le bandeau
     dédié et l'enrichissement de `DaemonStatus.DownReason` relèvent de **6.3b** ;
     en 6.3a l'alerte transite par le canal standard et apparaît déjà dans Security
     sans travail UI.

  Aucun secret ne vivant dans ce fichier, repartir des défauts est sûr.
- **Erreur d'I/O non réparable au boot** (dossier support inaccessible, disque
  plein, permissions refusées) : le ré-seed ne peut rien y faire → le daemon
  **exit avec un log d'erreur explicite**. Seul refus de boot légitime.
- **`config.set` invalide** : validation échoue → erreur RPC, **fichier inchangé**,
  rien d'appliqué (atomicité : valider le `Config` candidat complet avant tout
  effet de bord).
- **Échec d'écriture du backup** : l'écriture principale est **abandonnée** (on ne
  persiste pas sans avoir sécurisé l'état précédent) → erreur RPC.
- **Aucun secret** dans les logs/erreurs (ce fichier n'en contient pas, mais les
  messages d'erreur ne doivent pas dumper de valeurs de hosts inutilement).

## 10. Tests

`ConfigStore` (nouveaux tests) :
- Seed-si-absent : démarrage sans fichier → `config.json` créé == `Config.default`.
- Round-trip `config.set` → `config.get` (scalaire hot et scalaire restart).
- Rejet d'un `config.set` invalide → fichier intact, rien appliqué.
- Clé inconnue rejetée.
- Backup créé avant save ; **rotation au cap** (`max_count` respecté, plus anciens
  purgés) ; `max_count` modifié à chaud pris en compte au save suivant.
- Permissions 0600 sur le fichier et les backups.
- `config.json` corrompu au boot → backup du fichier corrompu + ré-seed sur défauts
  + `Alert` `severity: .high` émise (visible dans le ring/`events.query`) + daemon
  up + substitution active. Distinct du cas I/O non réparable (dossier
  inaccessible) → exit attendu.
- Hosts : `rule.add` → `origin: user` ; `rule.delete` d'un `origin: default` →
  rejet (`hostProtected`) ; `rule.delete` d'un `user` → OK ; `list` renvoie
  `origin` ; `created_at` présent.

Portage : les tests existants de `RuntimeRulesStore` pertinents sont transférés sur
`ConfigStore`. Intégration : `reload()` après édition manuelle du fichier
ré-applique (host ajouté hors-bande visible après reload).

## 12. Critères de réussite (DoD 6.3a)

1. `swift build` + `swift test` + `swift-format` verts ; CI verte (gate xcodebuild macos-15).
2. Démarrage à froid (aucun `config.json`) → seed correct, daemon opérationnel.
3. `iris config set security.on_exfil_attempt block_only` → appliqué à chaud
   (visible dans `iris config get`), persisté, backup créé.
4. `iris config set broker.listen 127.0.0.1:9999` → persisté + `requires_restart`.
5. `iris config set backups.max_count 3` puis ≥ 4 saves → ≤ 3 backups conservés.
6. `iris rule add` (→ `origin: user`) / `rule rm` reflétés dans `config.json`
   (`hosts`), substitution inchangée ; `rule rm api.anthropic.com` (origin
   `default`) → **rejeté** (`hostProtected`).
7. `config.json` corrompu au boot → daemon démarre en **mode dégradé** (ré-seed
   défauts + `Alert` `high` visible), substitution active, fichier corrompu sauvé
   en backup.
8. Aucune référence résiduelle à TOML/TOMLKit/`runtime-rules.json` dans le code.
9. Smoke poste : substitution `claude` toujours fonctionnelle après refonte.
