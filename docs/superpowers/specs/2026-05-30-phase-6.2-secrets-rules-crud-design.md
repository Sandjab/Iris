# Phase 6.2 — Menu bar app : Secrets + Rules CRUD

**Date** : 2026-05-30
**Status** : design approuvé (brainstorming), à transformer en plan d'implémentation
**Phase précédente** : 6.1b (menu-bar read-only Overview/Logs/Security + notifs, merged `28e8164`)
**Branche cible** : `feat/phase-6.2-secrets-rules-crud`
**Contexte 6.x** : `docs/superpowers/specs/2026-05-28-phase-6.1-menubar-readonly-design.md`

## 1. Objectif et portée

La Phase 6.1 a livré le consommateur read-only des events (Overview / Logs / Security + notifications). La Phase 6.2 ajoute la **gestion** des secrets et des règles MITM directement depuis le popover, en CRUD complet, conformément à SPECS §15.2 (onglets 4 Secrets et 5 Rules).

**6.2 livre :**

1. **Onglet Secrets** : liste + Add / Edit (allowed_hosts) / Rotate (nouvelle valeur) / Delete.
2. **Onglet Rules** (hosts MITM) : liste + Add (validation DNS) / Delete (confirmation) + bandeau d'avertissement de confiance.
3. Segmented control porté de 3 à **5 onglets**, présentation **icône + texte** (SF Symbols).
4. Formulaires **inline** dans le popover (navigation « ‹ Retour », pas de sheet ni de fenêtre séparée).
5. Extension testable de `IrisAppCore` (protocole `AdminCalling`, état `AppModel`, validation de formulaire pure).
6. Smoke checklist exhaustive vérifiée manuellement.

**Aucun changement daemon.** Tous les RPC de mutation sont déjà mergés (Phase 3 pour `secret.*`, Phase 4.x pour `rule.*`). La 6.2 est purement app-side.

**Hors scope (différé) :**

## 2. Décisions de design

Quatre décisions prises pendant le brainstorming (dont trois via maquettes navigateur), figées ici comme contraintes du plan.

### 2.1 — Périmètre : CRUD Secrets + Rules uniquement, Quarantine différé

SPECS §15.2.3 décrit une action « Quarantine secret » sur l'onglet Security. Contrairement au CRUD, elle n'a **aucun RPC daemon** : il faudrait un nouveau `AdminMethod`, un état daemon (set de secrets désactivés, avec question de persistance au reboot), et un branchement dans le chemin chaud de substitution (`PlaceholderEngine`/`ExfilRuleEngine`). Cela ferait passer la 6.2 de « UI pure sur RPC existants » à « UI + feature daemon touchant le hot path », avec tests de régression du scoping. **Différé** pour garder 6.2 mergeable en une PR à surface de review maîtrisée (cohérent avec la découpe 6.x : 1 sous-phase = 1 PR démontrable).

### 2.2 — Présentation des formulaires : inline dans le popover

Add / Edit / Rotate ne sont **ni un sheet SwiftUI ni un NSPanel séparé**, mais un **formulaire inline** : le clic sur une action bascule le contenu de l'onglet vers le formulaire, avec un header « ‹ Retour ».

**Justification** : le popover est `.applicationDefined` avec un monitor global (`NSEvent.addGlobalMonitorForEvents`) qui le ferme à tout clic hors de l'app (établi en 6.1b). Un sheet SwiftUI hébergé dans un NSPopover a des quirks de taille/détachement connus et serait exigu dans 480×600. Un NSPanel séparé (suggéré par la lettre de SPECS §15.2.4 pour Rotate) demande de la glue AppKit (lifecycle de fenêtre, suspension du monitor pendant l'ouverture, pont state à travers deux fenêtres). L'inline évite toute gestion de fenêtre/sheet, n'a aucun conflit avec le monitor, et reste cohérent avec « popover-only » (la full window est différée 6.3). L'écart vs la mention littérale « NSPanel » de SPECS est cosmétique et justifié.

### 2.3 — Segmented control : 5 onglets, icône + texte

Le segmented control passe de 3 à 5 segments (Overview / Logs / Security / Secrets / Rules), présentés **icône SF Symbol au-dessus du label** plutôt qu'en texte seul.

**Justification** : à 480 px, 5 labels texte rendent « Overview » et « Security » serrés (risque de troncature avec les réglages d'accessibilité de taille de texte). L'icône + texte est plus lisible, plus idiomatique macOS, et robuste à l'agrandissement. Léger écart cosmétique vs « segmented control » texte de SPECS §15.2. Icônes pressenties : `chart.bar` (Overview), `list.bullet` (Logs), `exclamationmark.shield` (Security), `key` (Secrets), `network` (Rules) — à confirmer au plan.

### 2.4 — Mutations : RPC puis re-fetch, jamais d'optimistic update

Toute mutation (`add`/`update`/`rotate`/`delete`) appelle le RPC, attend l'ack, **puis re-fetche la liste** (`secret.list` / `rule.list`) avant de publier le nouvel état. Pas de mise à jour optimiste de `AppModel.secrets`/`.rules`.

**Justification** : SPECS §15.4 « Mutations always go through admin RPC; UI updates after RPC ack to ensure consistency ». Le re-fetch garantit que l'UI reflète l'état autoritaire du daemon (y compris `created_at`/`usage_count` calculés daemon-side) sans dupliquer la logique de mutation côté app.

Le partage établi en 6.1 est conservé : **logique testable dans `IrisAppCore` (SwiftPM, couvert par `swift test` + CI macos)**, coquille UI mince dans `IrisApp` (Xcode, smoke manuel).

### 3.2 `IrisAppCore` — extensions

**`AdminCalling`** (protocole `Sendable`) — +8 méthodes, signatures calquées sur le contrat RPC existant (`Sources/IrisKit/IPC/AdminProtocol.swift`) :

```swift
func listSecrets() async throws -> [Secret]
func addSecret(name: String, allowedHosts: [String], value: Data) async throws -> Secret
func updateSecret(name: String, allowedHosts: [String]) async throws -> Secret
func rotateSecret(name: String, value: Data) async throws -> Secret
func deleteSecret(name: String) async throws
func listRules() async throws -> [MITMRule]
func addRule(host: String) async throws -> MITMRule
func deleteRule(host: String) async throws
```

La conformance dans `IrisKitConformances.swift` reprend verbatim les call-sites CLI canoniques (`SecretCommands.swift`, `RuleCommands.swift`) : `client.call(.secretAdd, params: SecretAddParams(...), returning: Secret.self)`, etc. Les valeurs de secret sont des `Data` (binary-safe, base64 sur le fil), jamais des `String` (chemin établi Phase 5 pour survivre aux secrets binaires).

**`AppModel`** — +2 `@Published` et leurs mutations :

```swift
@Published public var secrets: [Secret]   // trié par name
@Published public var rules: [MITMRule]    // trié par host

func addSecret(name:allowedHosts:value:via:) async throws    // RPC → re-fetch listSecrets → publish
func updateSecret(name:allowedHosts:via:) async throws
func rotateSecret(name:value:via:) async throws
func deleteSecret(name:via:) async throws
func addRule(host:via:) async throws
func deleteRule(host:via:) async throws
func refreshSecrets(via:) async throws / refreshRules(via:) async throws
```

Chaque mutation : appelle la méthode `AdminCalling` correspondante, puis `refresh*` (re-fetch + tri + publish). Erreurs propagées telles quelles (`DaemonUnreachable`, `JSONRPCError`) pour mapping côté UI.

**`SecretFormState` / `RuleFormState`** (struct/class pure, `@MainActor` si besoin de binding) — **toute la logique testée vit ici** :

- `SecretFormState` : champs `name`, `value`, `hostsInput`/`hosts: [String]`. Validation live via `Secret.validateName(_:)` (`^[a-zA-Z0-9_-]{1,64}$`) et `Secret.validateAllowedHosts(_:)` (validateur canonique daemon-side : non vide **et** chaque host valide). Mode `.add` / `.edit` (hosts only) / `.rotate` (value only). `canSubmit` : add = name valide **et** value non vide **et** ≥1 host valide ; edit = ≥1 host valide ; rotate = value non vide. Expose `validationError: String?`. Parse les tokens hosts (split, trim, dédup) ; un host vide ou malformé invalide la soumission.
- `RuleFormState` : champ `host`, validation live via `Secret.isValidHost(_:)`. `validationError` / `canSubmit`.

**`SyncCoordinator.bootstrap()`** : ajoute `listSecrets` + `listRules` au snapshot initial et au re-snapshot post-reconnexion (à côté de `status`/`stats`/`events.query`).

### 3.3 `IrisApp` — vues

**`PopoverView`** : `AppModel.Tab` gagne `.secrets` et `.rules` ; le Picker segmenté gagne 2 entrées (icône + texte via `Label`) ; le routeur gagne `case .secrets: SecretsTab()` / `case .rules: RulesTab()`.

**`SecretsTab`** : machine à 2 états internes (`@State route: .list | .form(Mode, Secret?)`).
- **Liste** : header avec bouton `+ Add`. Chaque secret = **row compacte** (480 px ne tient pas un tableau 5 colonnes) : ligne 1 `name` + chips `allowed_hosts` ; ligne 2 `created` · `last_used` · `usage_count` en gris. Actions de fin de row (icônes) : Edit · Rotate · Delete.
- **Formulaire inline** : header « ‹ Retour » + titre selon le mode, champs selon §3.2, bouton submit gaté par `canSubmit`, bandeau d'erreur RPC sous le bouton. Add : `TextField` name + `SecureField` value + champ à tokens hosts avec **autocomplétion depuis `model.rules`** (whitelist MITM) + texte libre (SPECS §15.2.4). Edit : hosts only. Rotate : `SecureField` value only.
- **Delete** : `.confirmationDialog` destructif (mention : supprime aussi l'item Keychain via le daemon).

**`RulesTab`** :
- **Liste** : header avec champ `+ Add host` (validation DNS live). Chaque rule = row `host` + badge `source` (`toml`/`runtime`). Les rules `source == .toml` sont **non supprimables** (bouton Delete masqué + tooltip « définie dans config.toml ») — le daemon renvoie `-32010 ruleProtected` (Phase 4.x).
- **Add** : submit → `addRule` → re-fetch. **Delete** (runtime only) : `.confirmationDialog` → `deleteRule`.
- **Bandeau** (SPECS §15.2.5, traduit) : « Ajouter un host signifie que le trafic vers ce host peut recevoir des substitutions de placeholders. Assure-toi qu'aucun secret n'autorise des hosts que tu ne contrôles pas. »

### 3.4 Flow données

```
SecretsTab/RulesTab ──action──> AppModel.{add,update,rotate,delete}*
        │                              │
        │                         AdminCalling (RPC) ──> daemon ──> Keychain / runtime-rules.json
        │                              │
        │                         re-fetch list*  <──── ack
        │                              │
        └───────── @Published ◀────────┘  (secrets / rules republiés, UI rerendue)
```

La valeur de secret (`Data`) ne transite que app→daemon via le socket UDS 0600 ; l'app n'accède **jamais** au Keychain (SPECS §12.3). Elle n'est jamais relue ni affichée.

## 4. Edge cases et error handling

| Situation | Comportement |
|---|---|
| **Daemon down pendant une mutation** | `AdminCalling` lève `DaemonUnreachable` → bandeau d'erreur dans le formulaire, on **reste** sur le formulaire (saisie préservée). |
| **Nom de secret déjà pris** | RPC `secret.add` rejette (erreur daemon : nom déjà présent) → message mappé sous le champ name. Code exact à relever au plan. |
| **Host invalide rejeté côté daemon** | `rule.add` renvoie `-32602` (validation daemon) malgré un `isValidHost` client OK (regex légèrement divergentes) → bandeau d'erreur. Le daemon reste l'autorité. |
| **Delete d'une rule TOML** | Bouton absent ; si tenté quand même, `-32010 ruleProtected` → message, jamais d'écrasement silencieux. |
| **Changement via CLI `iris` en parallèle** | Pas de SSE pour secrets/rules. Rattrapé au prochain bootstrap / ouverture d'onglet (re-fetch). Pas de polling périodique. |
| **allowed_hosts vide à l'Add / Edit** | **Bloqué** : `canSubmit == false`. Le daemon rejette les hosts vides — `KeychainSecretStore.add`/`.update` appellent `Secret.validateAllowedHosts`, qui lève `invalidAllowedHosts` sur `[]`. L'app exige ≥1 host valide pour ne pas envoyer un RPC voué à l'échec. (La notation `[--allowed-hosts]` de SPECS §4.3 est trompeuse : syntaxiquement optionnelle en argparse, mais le daemon l'impose ; la CLI échoue aussi sans host.) |
| **Valeur vide à Add/Rotate** | `canSubmit == false`. |
| **Secret supprimé pendant qu'on est sur son formulaire Edit/Rotate** | Le RPC échoue (not found) → bandeau d'erreur + retour liste re-fetchée. |

## 5. Sync : rafraîchissement secrets/rules

SSE (Phase 3) ne transporte que des `Event`, pas les changements de secrets/rules. Stratégie : **fetch au bootstrap + après chaque mutation + à l'ouverture de l'onglet** (`.task` SwiftUI). **Pas de polling périodique** : ces données changent rarement et seulement via cette app ou la CLI ; un poll 30 s serait du bruit pour un cas marginal. Un changement CLI concurrent est rattrapé au prochain affichage de l'onglet.

## 6. Testing strategy

### 6.1 Tests automatisés (`IrisAppCoreTests`)

- **`SecretFormState`** : table-driven sur validation name (regex valide/invalide/longueur), value (vide/non vide/binaire), hosts (parsing tokens, dédup, host invalide), `canSubmit` par mode (add/edit/rotate), `validationError` exact.
- **`RuleFormState`** : validation host DNS (valide, invalide, vide, > 253, espaces, path).
- **`AppModel` CRUD** : avec `FakeAdminCalling` étendu — chaque mutation appelle le bon RPC puis re-fetche (vérifier l'ordre RPC→list) ; propagation des erreurs ; tri stable de `secrets`/`rules`.
- **Invariant verrouillé** : `testSecretFormNeverRetainsOrLogsValue` — la valeur saisie n'apparaît dans aucune description/log/état publié (CLAUDE.md §6.1, I2). Le `FakeAdminCalling` capture la `Data` reçue mais aucune sérialisation d'`AppModel`/event ne doit la contenir.

Pas de tests SwiftUI Views (cohérent 6.1 : glue UI = smoke manuel).

### 6.2 CI

- Step `swift build + swift test + swift-format lint` : étendu silencieusement par les nouveaux fichiers `IrisAppCore` + `IrisAppCoreTests`.
- `swift-format lint --strict --recursive Sources Tests IrisApp` : couvre les nouvelles vues `IrisApp`.
- **Step `xcodebuild`** : **promu de `continue-on-error` à gate dur**, avec bump runner `macos-14 → macos-15` (Xcode 16 requis pour `objectVersion 77` du projet). *Recommandé ; repli possible : garder `continue-on-error` en 6.2 si le runner macos-15 pose problème, et promouvoir ultérieurement.*

### 6.3 Smoke checklist PR (cases à cocher avant merge)

- [ ] Onglet Secrets : liste affiche name + allowed_hosts + created/last_used/usage_count corrects.
- [ ] Add : validation live du nom (regex), `SecureField` value, tokens hosts avec autocomplétion depuis les rules MITM. Submit → secret apparaît dans la liste.
- [ ] Edit : seul allowed_hosts éditable ; submit → liste reflète les nouveaux hosts.
- [ ] Rotate : `SecureField` value only ; submit → pas d'erreur, usage inchangé.
- [ ] Delete : confirmation destructive ; submit → secret retiré de la liste ET du Keychain (vérifiable via `iris secret list`).
- [ ] Onglet Rules : liste host + badge source ; rule TOML non supprimable (Delete masqué).
- [ ] Add rule : validation DNS live ; submit → host apparaît, badge `runtime`. Bandeau d'avertissement visible.
- [ ] Delete rule runtime : confirmation → host retiré.
- [ ] Mutation avec daemon coupé → bandeau d'erreur, saisie préservée, pas de crash.
- [ ] Changement via CLI `iris secret add` puis ré-ouverture de l'onglet → liste rafraîchie.
- [ ] Aucune valeur de secret visible nulle part (UI, logs, notifs) — invariant §6.1.
- [ ] 5 segments icône+texte lisibles, onglet sélectionné persisté entre relances.

## 8. Limitations connues / différé explicite

- **Quarantine secret** (SPECS §15.2.3) : nécessite un RPC daemon `secret.set_enabled` (ou équivalent) + état + skip substitution + tests de régression scoping → 6.2.x.
- **Divergence regex host** : `Secret.isValidHost` (client, public) ≠ `RuntimeRulesStore.hostPattern` (daemon). Validation client = feedback live ; daemon = autorité. Alignement éventuel en cleanup, hors-scope 6.2.
- **Autocomplétion hosts** depuis les rules MITM seulement (pas d'historique ni de suggestions externes).
- **Pas de polling** secrets/rules : changements CLI concurrents rattrapés à l'ouverture d'onglet.
